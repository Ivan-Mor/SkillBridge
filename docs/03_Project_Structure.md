# Директория apps

Все бизнес-модули располагаются внутри директории `apps`.

```
apps/
│
├── accounts/
├── profiles/
├── skills/
├── offers/
├── reviews/
├── notifications/
└── common/
```

Каждое приложение является независимым модулем системы.

Это позволяет:

- уменьшить связанность компонентов;
- упростить тестирование;
- ускорить разработку;
- повторно использовать код;
- масштабировать проект.

# Назначение приложений

## accounts

Отвечает за:

- регистрацию;
- авторизацию;
- восстановление пароля;
- работу с пользователями;
- управление Custom User Model.

---

## profiles

Отвечает за пользовательский профиль.

Содержит:

- информацию о пользователе;
- фотографию;
- описание;
- опыт;
- дополнительные сведения.

---

## skills

Основной модуль проекта.

Содержит:

- навыки пользователя;
- категории навыков;
- уровни владения;
- список изучаемых навыков.

---

## offers

Отвечает за обмен знаниями между пользователями.

Содержит:

- предложения;
- статусы;
- подтверждение обмена;
- завершение обучения.

---

## reviews

Система отзывов.

Позволяет:

- оставлять оценки;
- публиковать отзывы;
- рассчитывать рейтинг пользователя.

---

## notifications

Отвечает за уведомления пользователей.

Используется для:

- новых предложений;
- изменения статусов;
- новых отзывов;
- системных сообщений.

---

## common

Общие компоненты проекта.

Используется всеми приложениями.

Содержит:

- константы;
- утилиты;
- миксины;
- валидаторы;
- общие permissions.

# Почему используется Service Layer

Бизнес-логика не должна находиться во View.

View должна:

- получить запрос;
- вызвать сервис;
- вернуть ответ.

Пример:

```
View
    ↓
Service
    ↓
Selector
    ↓
Model
```

Такое разделение делает код:

- проще;
- понятнее;
- легче тестируемым;
- пригодным для повторного использования.

---

# Почему используются Selectors

Чтение данных отделяется от бизнес-логики.

Selectors отвечают только за:

- получение объектов;
- фильтрацию;
- сортировку;
- сложные SQL-запросы;
- оптимизацию ORM.

Это позволяет избежать дублирования запросов в разных частях проекта.

# Принципы организации проекта

При разработке SkillBridge используются следующие принципы:

- модульность;
- разделение ответственности (Separation of Concerns);
- единая структура приложений;
- минимальная связанность компонентов;
- высокая читаемость кода;
- масштабируемость;
- повторное использование компонентов.

Все приложения придерживаются единого стандарта организации файлов, что облегчает навигацию по проекту и ускоряет подключение новых разработчиков.

---

# Масштабируемость

Выбранная структура позволяет без значительных изменений добавлять новые приложения.

Например:

```
apps/
│
├── chat/
├── recommendations/
├── api/
├── analytics/
└── payments/
```

Каждый новый модуль интегрируется в проект без нарушения существующей архитектуры.

---

# Заключение

Структура проекта SkillBridge разработана с учётом современных практик Django-разработки и ориентирована на долгосрочную поддержку. Единая организация директорий, разделение бизнес-логики, использование Service Layer и модульный подход делают проект удобным для командной работы, тестирования и последующего масштабирования.
