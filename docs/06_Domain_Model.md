# Введение

Domain Model — это описание предметной области приложения.

Она определяет:

- основные сущности системы;
- отношения между ними;
- ограничения;
- бизнес-правила;
- жизненный цикл объектов;
- основные пользовательские сценарии.

В отличие от моделей Django, Domain Model описывает не программную реализацию, а логику предметной области.

Именно на основе данного документа впоследствии разрабатываются:

- модели базы данных;
- сервисы;
- формы;
- представления;
- API;
- автоматические тесты.

---

# Предметная область

SkillBridge представляет собой платформу обмена знаниями между пользователями.

Каждый зарегистрированный пользователь может одновременно выступать в двух ролях:

- преподаватель;
- ученик.

Вместо классической схемы

```
Преподаватель

↓

Ученик
```

используется модель взаимного обучения.

```
Пользователь A

умеет Python

хочет изучить Blender


⇅


Пользователь B

умеет Blender

хочет изучить Python
```

Таким образом, один пользователь способен обучать других одним навыкам и одновременно обучаться другим.

# User

## Назначение

Модель **User** является центральной сущностью всей системы.

Практически каждый объект приложения прямо или косвенно связан с пользователем.

Через User реализуются:

- регистрация;
- авторизация;
- управление аккаунтом;
- взаимодействие между пользователями;
- получение уведомлений;
- участие в обмене знаниями.

Без существования пользователя невозможна работа остальных модулей проекта.

---

## Ответственность

Модель отвечает исключительно за информацию, необходимую для функционирования учетной записи.

Она **не должна** хранить:

- биографию;
- описание;
- фотографию;
- навыки;
- рейтинг.

Подобная информация относится к другим сущностям предметной области.

Это соответствует принципу **Single Responsibility Principle**.

---

## Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| id | BigAutoField | Первичный ключ |
| username | CharField | Имя пользователя |
| email | EmailField | Адрес электронной почты |
| password | CharField | Хэш пароля |
| is_active | Boolean | Статус аккаунта |
| is_staff | Boolean | Доступ в административную панель |
| date_joined | DateTime | Дата регистрации |
| last_login | DateTime | Последняя авторизация |

---

## Связи

```
User

├── 1 → 1 Profile

├── 1 → N UserSkill

├── 1 → N Sent Offers

├── 1 → N Received Offers

├── 1 → N Reviews

└── 1 → N Notifications
```

---

## Бизнес-правила

Для модели User устанавливаются следующие правила.

### Регистрация

Пользователь может зарегистрироваться только один раз с использованием уникального адреса электронной почты.

---

### Авторизация

Вход выполняется по электронной почте и паролю.

Пароль хранится исключительно в виде безопасного хэша.

---

### Уникальность

Должны быть уникальными:

- username;
- email.

Создание двух пользователей с одинаковыми значениями запрещено.

---

### Активность

Неактивный пользователь не может:

- входить в систему;
- отправлять предложения;
- получать предложения;
- оставлять отзывы.

---

### Удаление

Физическое удаление пользователя не рекомендуется.

Предпочтительно использовать механизм деактивации аккаунта.

Это позволяет сохранить:

- отзывы;
- историю обменов;
- статистику;
- журнал действий.

---

## Основные операции

Пользователь может:

- зарегистрироваться;
- войти в систему;
- изменить пароль;
- восстановить пароль;
- изменить электронную почту;
- выйти из системы;
- удалить аккаунт (деактивация).

---

## Жизненный цикл

```
Регистрация

↓

Создание аккаунта

↓

Авторизация

↓

Использование платформы

↓

Редактирование профиля

↓

Обмен знаниями

↓

Деактивация
```

---

# Profile

## Назначение

Profile содержит расширенную информацию о пользователе.

Разделение данных между User и Profile позволяет избежать перегруженности модели пользователя и упрощает сопровождение системы.

Подход соответствует распространённой практике разработки на Django.

---

## Почему Profile выделен отдельно

Использование отдельной модели Profile имеет ряд преимуществ:

- уменьшение размера таблицы пользователей;
- разделение ответственности;
- возможность независимого расширения профиля;
- снижение количества изменений в модели User.

---

## Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| user | OneToOneField | Связь с User |
| avatar | ImageField | Фотография пользователя |
| first_name | CharField | Имя |
| last_name | CharField | Фамилия |
| city | CharField | Город |
| bio | TextField | Описание пользователя |
| experience | TextField | Опыт |
| created_at | DateTime | Дата создания |
| updated_at | DateTime | Последнее изменение |

---

## Связи

```
User

↓

Profile
```

Каждый пользователь может иметь только один профиль.

Каждый профиль принадлежит только одному пользователю.

---

## Бизнес-правила

После регистрации профиль создаётся автоматически.

Профиль может существовать только при наличии связанного пользователя.

Удаление пользователя приводит к удалению профиля.

---

## Основные операции

Пользователь может:

- загрузить фотографию;
- изменить личные данные;
- изменить описание;
- указать опыт;
- обновить информацию о себе.

---

## Жизненный цикл

```
Создание пользователя

↓

Автоматическое создание Profile

↓

Заполнение данных

↓

Редактирование

↓

Использование при поиске

↓

Обновление информации
```

---

## Ограничения

Для профиля устанавливаются следующие ограничения.

- Один пользователь — один профиль.
- Аватар является необязательным.
- Размер загружаемого изображения ограничивается настройками приложения.
- Поля описания допускают пустые значения до момента заполнения пользователем.

# Skill

## Назначение

Модель **Skill** представляет собой справочник всех навыков, существующих на платформе.

Она является единым источником данных и используется всеми пользователями.

Навык существует независимо от конкретного пользователя.

Например:

- Python
- Django
- PostgreSQL
- Blender
- Figma
- Photoshop

Каждый навык создаётся один раз и затем используется многими пользователями.

---

## Основная задача модели

Skill отвечает исключительно за описание навыка.

Модель **не хранит** информацию:

- кто владеет навыком;
- кто хочет его изучить;
- уровень владения;
- опыт пользователя.

Эти данные относятся к модели **UserSkill**.

---

## Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| id | BigAutoField | Первичный ключ |
| name | CharField | Название навыка |
| slug | SlugField | URL-идентификатор |
| description | TextField | Краткое описание |
| is_active | Boolean | Используется ли навык |
| created_at | DateTime | Дата создания |
| updated_at | DateTime | Последнее изменение |

---

## Связи

```
Skill

↓

UserSkill

↓

User
```

Один навык может принадлежать большому количеству пользователей.

---

## Бизнес-правила

### Уникальность

Название навыка должно быть уникальным.

Например:

Допустимо

```
Python
```

Недопустимо

```
Python

Python
```

---

### Доступность

Удаление навыков не рекомендуется.

Если навык перестал использоваться, он переводится в состояние:

```
is_active = False
```

Это позволяет сохранить историю пользователей.

---

### Использование

Навык может использоваться одновременно:

- преподавателями;
- учениками.

---

## Основные операции

Администратор может:

- создать навык;
- изменить описание;
- отключить навык.

Пользователь может:

- выбрать навык;
- искать пользователей по навыку;
- использовать навык при создании предложения обмена.

---

## Жизненный цикл

```
Создание навыка

↓

Использование пользователями

↓

Редактирование

↓

Архивация
```

# UserSkill

## Назначение

UserSkill является промежуточной моделью между User и Skill.

Именно она содержит информацию о том:

- каким навыком владеет пользователь;
- насколько хорошо владеет;
- хочет ли обучать;
- хочет ли изучать.

Без данной модели невозможно реализовать основную идею проекта.

---

## Почему используется отдельная модель

Связь Many-to-Many через промежуточную модель позволяет хранить дополнительные данные.

Если использовать обычное ManyToManyField, будет невозможно сохранить:

- уровень владения;
- статус;
- дату добавления;
- комментарий пользователя.

Поэтому используется модель-посредник.

---

## Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| user | ForeignKey | Пользователь |
| skill | ForeignKey | Навык |
| role | CharField | Роль навыка |
| level | CharField | Уровень владения |
| comment | TextField | Комментарий |
| created_at | DateTime | Дата добавления |

---

## Возможные роли

Пользователь может указать одну из ролей.

| Значение | Назначение |
|-----------|------------|
| Teaching | Готов обучать |
| Learning | Хочет изучить |
| Both | И обучает, и изучает |

---

## Возможные уровни

Предлагается использовать следующие уровни.

| Уровень | Описание |
|----------|----------|
| Beginner | Начинающий |
| Elementary | Базовый |
| Intermediate | Средний |
| Advanced | Продвинутый |
| Expert | Эксперт |

---

## Связи

```
User

↓

UserSkill

↓

Skill
```

Таким образом реализуется связь

```
Many To Many
```

между пользователями и навыками.

---

## Бизнес-правила

Один пользователь может иметь множество навыков.

Один навык может принадлежать множеству пользователей.

---

Для одного пользователя запрещается создание двух одинаковых записей.

Например

Недопустимо

```
Python

Python
```

Для этого используется ограничение

```
UniqueConstraint

(user, skill)
```

---

Пользователь может изменить:

- уровень;
- комментарий;
- роль.

При этом сама запись остаётся неизменной.

---

## Основные операции

Пользователь может:

- добавить навык;
- удалить навык;
- изменить уровень;
- изменить комментарий;
- изменить роль.

---

## Жизненный цикл

```
Добавление навыка

↓

Выбор уровня

↓

Выбор роли

↓

Использование при поиске

↓

Редактирование

↓

Удаление
```

---

# Использование UserSkill

При поиске преподавателей система анализирует только записи

```
role = Teaching
```

При поиске учеников

```
role = Learning
```

Если пользователь указал

```
Both
```

то он участвует сразу в двух видах поиска.

---

# Поиск пользователей

Пример поиска.

Пользователь хочет изучить

```
Python
```

Алгоритм выполняет:

```
Поиск навыка

↓

Получение UserSkill

↓

Фильтрация

role = Teaching

↓

Получение пользователей

↓

Сортировка

↓

Вывод результата
```

---

# Основные ограничения

Для UserSkill действуют следующие правила.

- Один пользователь может добавить навык только один раз.
- Один навык может использоваться тысячами пользователей.
- Уровень владения выбирается только из фиксированного списка.
- Роль выбирается только из фиксированного списка.
- При удалении пользователя удаляются все его навыки.
- При отключении навыка существующие записи сохраняются.

---

# Почему выбрана именно такая модель

Использование промежуточной модели предоставляет несколько преимуществ.

- хранение дополнительных данных;
- расширяемость;
- удобная фильтрация;
- простая работа через ORM;
- возможность добавления рейтингов, сертификатов и опыта без изменения структуры базы данных.

Именно UserSkill является центральной моделью предметной области SkillBridge, так как соединяет пользователей и навыки, формируя основу механизма поиска и обмена знаниями.

# Offer

## Назначение

Модель **Offer** представляет собой предложение обмена знаниями между двумя пользователями.

Offer является центральной бизнес-сущностью проекта.

Именно она описывает сам процесс взаимодействия пользователей и связывает воедино все остальные сущности системы.

Без Offer платформа превращается в обычный каталог пользователей.

---

# Основная идея

Каждое предложение создаётся одним пользователем и направляется другому.

Например

```
Иван

умеет Python

хочет изучить Blender

↓

создаёт предложение

↓

Марии

умеет Blender

хочет изучить Python
```

После принятия предложения пользователи начинают обмен знаниями.

---

# Ответственность модели

Offer отвечает исключительно за процесс обмена.

Она хранит информацию:

- кто отправил предложение;
- кому отправлено предложение;
- какой навык предлагается;
- какой навык пользователь хочет изучить;
- текущий статус предложения;
- дату создания;
- дату завершения.

---

# Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| id | BigAutoField | Первичный ключ |
| sender | ForeignKey(User) | Отправитель |
| receiver | ForeignKey(User) | Получатель |
| teach_skill | ForeignKey(Skill) | Навык, которому готов обучать отправитель |
| learn_skill | ForeignKey(Skill) | Навык, который хочет изучить отправитель |
| message | TextField | Сопроводительное сообщение |
| status | CharField | Статус предложения |
| created_at | DateTime | Дата создания |
| updated_at | DateTime | Последнее изменение |
| completed_at | DateTime | Дата завершения |

---

# Связи

```
User

↓

Offer

↓

User

↓

Review
```

Каждое предложение связывает двух пользователей.

После успешного завершения обмена появляется возможность оставить отзыв.

---

# Жизненный цикл предложения

```
Создание

↓

Ожидание ответа

↓

Принятие

↓

Обучение

↓

Завершение

↓

Отзывы

↓

Архив
```

---

# Статусы предложения

Предлагается использовать следующие статусы.

| Статус | Описание |
|----------|----------|
| Pending | Ожидает ответа |
| Accepted | Принято |
| Declined | Отклонено |
| Cancelled | Отменено отправителем |
| Completed | Обмен завершён |

---

# Диаграмма переходов

```
                Pending
               /   |    \
              /    |     \
             /     |      \
            ▼      ▼       ▼
      Accepted Declined Cancelled
           │
           ▼
      Completed
```

Возврат к предыдущим статусам невозможен.

---

# Бизнес-правила

## Создание

Предложение может создать только авторизованный пользователь.

---

## Самому себе

Запрещается создавать предложение самому себе.

```
sender == receiver

↓

Ошибка
```

---

## Повторные предложения

Нельзя создать второе активное предложение тому же пользователю по той же паре навыков.

Для этого используется ограничение уникальности на уровне бизнес-логики.

---

## Проверка навыков

Перед созданием предложения система проверяет:

- отправитель действительно умеет преподавать выбранный навык;
- отправитель действительно хочет изучить указанный навык;
- получатель обладает требуемым навыком;
- получатель заинтересован в изучении навыка отправителя.

Только после успешной проверки предложение может быть создано.

---

## Отмена

Отправитель может отменить предложение только до момента принятия.

```
Pending

↓

Cancelled
```

После принятия отмена невозможна.

---

## Принятие

Получатель может:

- принять предложение;
- отклонить предложение.

После принятия статус меняется на

```
Accepted
```

---

## Завершение

Завершить можно только принятое предложение.

```
Accepted

↓

Completed
```

---

## Удаление

Физическое удаление предложений не выполняется.

Все предложения сохраняются в истории пользователя.

Пользователь может:

- создать предложение;
- просмотреть входящие предложения;
- просмотреть отправленные предложения;
- принять предложение;
- отклонить предложение;
- отменить предложение;
- завершить обмен.

✓ пользователи различны

✓ оба пользователя активны

# Ограничения

- Предложение всегда имеет одного отправителя.
- Предложение всегда имеет одного получателя.
- Предложение нельзя передать третьему лицу.
- После завершения статус изменить нельзя.
- Завершённое предложение нельзя повторно открыть.
- Каждое предложение имеет собственную историю изменений.

# Review

## Назначение

Модель **Review** представляет отзыв одного пользователя о другом после завершённого обмена знаниями.

Отзывы формируют систему доверия внутри SkillBridge.

Они позволяют пользователям оценивать качество взаимодействия, уровень объяснения материала, ответственность и общий опыт обучения.

---

## Ответственность модели

Review хранит информацию:

- кто оставил отзыв;
- кому оставлен отзыв;
- к какому обмену относится отзыв;
- какую оценку поставил пользователь;
- какой текстовый комментарий был оставлен;
- когда отзыв был создан.

Review не отвечает за расчёт рейтинга напрямую. Расчёт среднего рейтинга может выполняться отдельным сервисом или селектором.

---

## Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| id | BigAutoField | Первичный ключ |
| offer | ForeignKey(Offer) | Завершённое предложение обмена |
| author | ForeignKey(User) | Автор отзыва |
| recipient | ForeignKey(User) | Получатель отзыва |
| rating | PositiveSmallIntegerField | Оценка пользователя |
| text | TextField | Текст отзыва |
| created_at | DateTime | Дата создания |
| updated_at | DateTime | Последнее изменение |

---

## Связи

```text
Offer

↓

Review

↓

User
```

Один завершённый обмен может иметь отзывы от обеих сторон.

Например:

```text
Offer #1

├── Review от отправителя получателю
└── Review от получателя отправителю
```

---

## Бизнес-правила

### Отзыв только после завершения

Отзыв можно оставить только по предложению со статусом:

```text
Completed
```

Если предложение находится в статусе `Pending`, `Accepted`, `Declined` или `Cancelled`, отзыв создать нельзя.

---

### Только участники обмена

Оставить отзыв может только пользователь, который участвовал в конкретном предложении.

Недопустимо:

```text
Пользователь C

↓

оставляет отзыв по обмену между A и B
```

---

### Один отзыв от одного участника

Один пользователь может оставить только один отзыв в рамках одного завершённого обмена.

Для этого используется ограничение:

```text
UniqueConstraint(author, offer)
```

---

### Нельзя оставить отзыв самому себе

Автор и получатель отзыва должны быть разными пользователями.

```text
author != recipient
```

---

### Оценка ограничена диапазоном

Рейтинг должен находиться в диапазоне:

```text
1 ≤ rating ≤ 5
```

---

## Основные операции

Пользователь может:

- оставить отзыв;
- просмотреть отзывы другого пользователя;
- просмотреть свои полученные отзывы;
- просмотреть отзывы по конкретному обмену.

Редактирование отзывов можно ограничить или полностью запретить в зависимости от выбранной продуктовой логики.

---

## Жизненный цикл

```text
Завершение Offer

↓

Открытие возможности оставить отзыв

↓

Создание Review

↓

Обновление рейтинга пользователя

↓

Отображение в профиле
```

---

## Валидация

Перед созданием отзыва система проверяет:

- предложение существует;
- предложение завершено;
- автор является участником предложения;
- получатель является вторым участником предложения;
- автор ещё не оставлял отзыв по этому предложению;
- рейтинг находится в допустимом диапазоне;
- текст не превышает максимально допустимую длину.

---

## Использование в системе

Review используется для:

- отображения репутации пользователя;
- повышения доверия между участниками;
- сортировки пользователей по рейтингу;
- анализа качества взаимодействия;
- формирования истории обучения.

---

## Возможности дальнейшего развития

В будущем система отзывов может быть расширена.

Возможные улучшения:

- жалобы на отзывы;
- модерация отзывов;
- скрытие некорректных отзывов;
- ответы на отзывы;
- отдельные оценки по критериям;
- автоматический пересчёт рейтинга;
- бейджи за положительные отзывы.

# Notification

## Назначение

Модель **Notification** представляет системное уведомление для пользователя.

Уведомления помогают пользователю своевременно узнавать о важных событиях в системе.

Например:

- новое предложение обмена;
- принятие предложения;
- отклонение предложения;
- завершение обмена;
- новый отзыв;
- системное сообщение.

---

## Ответственность модели

Notification хранит информацию:

- кому адресовано уведомление;
- какой тип события произошёл;
- текст уведомления;
- прочитано ли уведомление;
- когда оно было создано;
- с каким объектом связано.

Notification не должна содержать сложную бизнес-логику. Создание уведомлений выполняется через сервисы или сигналы.

---

## Основные поля

| Поле | Тип | Описание |
|------|-----|----------|
| id | BigAutoField | Первичный ключ |
| recipient | ForeignKey(User) | Получатель уведомления |
| type | CharField | Тип уведомления |
| title | CharField | Заголовок |
| message | TextField | Текст уведомления |
| is_read | BooleanField | Прочитано ли уведомление |
| created_at | DateTime | Дата создания |
| read_at | DateTime | Дата прочтения |

---

## Возможные типы уведомлений

| Тип | Назначение |
|-----|------------|
| OfferCreated | Новое предложение обмена |
| OfferAccepted | Предложение принято |
| OfferDeclined | Предложение отклонено |
| OfferCancelled | Предложение отменено |
| OfferCompleted | Обмен завершён |
| ReviewCreated | Новый отзыв |
| System | Системное уведомление |

---

## Связи

```text
User

↓

Notification
```

Один пользователь может иметь множество уведомлений.

Каждое уведомление принадлежит только одному пользователю.

---

## Бизнес-правила

### Уведомление всегда имеет получателя

Нельзя создать уведомление без пользователя-получателя.

---

### Уведомление создаётся на значимое событие

Notification создаётся только при событиях, которые важны для пользователя.

Например:

```text
Offer created

↓

Notification for receiver
```

---

### Прочтение уведомления

Пользователь может отметить уведомление как прочитанное.

После этого:

```text
is_read = True
read_at = current_datetime
```

---

### История уведомлений сохраняется

Уведомления не удаляются сразу после прочтения.

Они могут отображаться в истории пользователя.

---

## Основные операции

Пользователь может:

- просмотреть список уведомлений;
- просмотреть только непрочитанные уведомления;
- отметить уведомление как прочитанное;
- отметить все уведомления как прочитанные.

---

## Жизненный цикл

```text
Событие в системе

↓

Создание Notification

↓

Отображение пользователю

↓

Прочтение

↓

Архив
```

---

## Примеры событий

### Создание предложения

```text

↓

создаёт Offer

↓

User B получает Notification
```

---

### Принятие предложения

```text

↓

↓

```

---

### Создание отзыва

```text

↓

```

---

## Валидация

Перед созданием уведомления система проверяет:

- получатель существует;
- получатель активен;
- тип уведомления допустим;
- сообщение не пустое;
- связанное событие действительно произошло.

---

## Использование в системе

Notification используется для повышения удобства работы с платформой.

Она помогает пользователю не пропустить:

- входящие предложения;
- изменение статусов;
- завершение обмена;
- появление отзывов.

# Взаимодействие Review и Notification

После создания отзыва система может автоматически создать уведомление получателю.

```text
Review created

↓

Notification created

↓

Recipient sees new review
```

Такой процесс может быть реализован через:

- Service Layer;
- Django Signals;
- Celery task.

На первом этапе предпочтительно использовать Service Layer, чтобы бизнес-логика оставалась явной и контролируемой.

# Заключение части

Модели **Review** и **Notification** дополняют основную бизнес-логику SkillBridge.

`Review` отвечает за доверие и репутацию пользователей.

`Notification` отвечает за своевременное информирование пользователей о важных событиях.

Вместе они делают платформу более живой, полезной и приближенной к реальному коммерческому веб-приложению.
