# Архитектурный подход

SkillBridge разрабатывается как монолитное веб-приложение на Django.

Несмотря на монолитную архитектуру, проект строится модульно. Каждый функциональный блок представляет собой независимое Django-приложение с чётко определённой зоной ответственности.

Подобный подход обеспечивает:

- простоту разработки;
- высокую скорость запуска проекта;
- лёгкость сопровождения;
- возможность постепенного масштабирования;
- минимальную связанность компонентов.

---

# Почему монолит?

На этапе разработки полноценный модульный монолит значительно эффективнее микросервисной архитектуры.

Причины выбора:

- меньше инфраструктурной сложности;
- проще развёртывание;
- единая база данных;
- единая авторизация;
- отсутствие сетевого взаимодействия между сервисами;
- проще тестирование;
- проще сопровождение небольшой командой.

При необходимости отдельные модули смогут быть выделены в самостоятельные сервисы без полного переписывания системы.

---

# Основные архитектурные принципы

При разработке проекта используются следующие принципы.

## Separation of Concerns (SoC)

Каждый слой системы отвечает только за собственную область ответственности.

Например:

- View не содержит бизнес-логики;
- Service не работает с HTTP;
- Selector не изменяет данные;
- Model не содержит сценариев работы приложения.

---

## Single Responsibility Principle (SRP)

Каждый модуль выполняет только одну задачу.

Например:

- сервис создаёт предложение обмена;
- селектор получает список предложений;
- форма валидирует данные;
- представление возвращает HTTP-ответ.

---

## DRY

Каждая бизнес-операция реализуется один раз и переиспользуется в разных местах приложения.

---

## KISS

Архитектура остаётся максимально понятной.

Мы избегаем избыточных абстракций и преждевременной оптимизации.

---

## Высокая связность и слабое зацепление

Каждое приложение отвечает за собственный функционал и минимально зависит от других модулей.

# Архитектура Django

Каждое приложение состоит из нескольких логических слоёв.

```
Request

↓

URL Dispatcher

↓

View

↓

Service

↓

Selector

↓

Model (ORM)

↓

PostgreSQL
```

Каждый слой отвечает только за собственную задачу.

---

# Ответственность слоёв

| Слой | Назначение |
|------|------------|
| URL | Маршрутизация запросов |
| View | Обработка HTTP-запросов и ответов |
| Form | Валидация пользовательского ввода |
| Service | Бизнес-логика |
| Selector | Получение данных из базы |
| Model | Описание структуры данных |
| ORM | Работа с PostgreSQL |

---

# View Layer

View является точкой входа в приложение.

View должна выполнять только следующие действия:

- принять запрос;
- проверить права доступа;
- вызвать форму;
- вызвать сервис;
- вернуть ответ.

View **не должна**:

- выполнять SQL-запросы;
- содержать бизнес-логику;
- создавать сложные ORM-запросы;
- принимать архитектурные решения.

Пример последовательности действий:

```
HTTP Request

↓

View

↓

Service

↓

HTTP Response
```

---

# Form Layer

Forms используются исключительно для обработки и проверки пользовательского ввода.

Основные задачи:

- валидация данных;
- преобразование типов;
- отображение ошибок;
- подготовка данных для передачи в Service Layer.

---

# Service Layer

Service Layer является центральным элементом архитектуры проекта.

Именно здесь располагается вся бизнес-логика приложения.

Например:

- регистрация пользователя;
- создание профиля;
- отправка предложения обмена;
- завершение обучения;
- публикация отзыва;
- расчёт рейтинга.

Преимущества:

- повторное использование кода;
- простота тестирования;
- независимость от HTTP;
- лёгкая интеграция с API или Celery.

---

## Пример взаимодействия

```
View

↓

OfferService.create_offer()

↓

OfferSelector.get_users()

↓

Offer.objects.create()
```

---

# Selector Layer

Selectors отвечают только за получение данных.

Они используются для:

- поиска пользователей;
- фильтрации;
- сортировки;
- оптимизации ORM;
- сложных JOIN-запросов;
- агрегаций.

Selectors никогда не изменяют данные.

---

# Model Layer

Модели являются отражением структуры базы данных.

Они содержат:

- описание таблиц;
- ограничения;
- связи;
- минимальную модельную логику.

Основная бизнес-логика находится в сервисах.

---

# Работа с PostgreSQL

Вся работа с базой выполняется через Django ORM.

Используются:

- Foreign Keys;
- OneToOne;
- ManyToMany;
- транзакции;
- индексы;
- ограничения целостности.

SQL-запросы не должны находиться во View.

---

# Redis

Redis используется как отдельный инфраструктурный компонент.

Планируется применение для:

- кэширования;
- хранения сессий;
- Celery Broker;
- очередей задач;
- ограничения частоты запросов (Rate Limiting).

# Основные архитектурные решения

| Решение | Причина |
|----------|----------|
| Модульный монолит | Простота разработки и сопровождения |
| Service Layer | Разделение бизнес-логики и HTTP |
| Selector Layer | Повторное использование запросов и оптимизация ORM |
| PostgreSQL | Надёжность, производительность и масштабируемость |
| Redis | Кэширование и фоновые задачи |
| Docker | Единое окружение разработки |
| Bootstrap | Быстрое создание адаптивного интерфейса |
| Git Flow | Удобная командная разработка |

# Заключение

Архитектура SkillBridge основана на принципах модульности, разделения ответственности и масштабируемости. Использование Service Layer и Selector Layer позволяет отделить бизнес-логику от представлений и доступа к данным, что делает код более понятным, тестируемым и удобным для сопровождения.

Выбранная архитектура соответствует современным практикам коммерческой Django-разработки и обеспечивает надёжную основу для дальнейшего развития проекта.
