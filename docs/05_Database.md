# Общий обзор

SkillBridge использует **PostgreSQL** в качестве основной системы управления базами данных.

PostgreSQL является промышленным стандартом для большинства Django-проектов и обеспечивает:

- высокую производительность;
- надёжность;
- поддержку транзакций;
- расширенные возможности индексирования;
- полнотекстовый поиск;
- JSONB;
- строгую целостность данных.

Все данные проекта хранятся в единой централизованной базе данных.

---

# Почему PostgreSQL

Выбор PostgreSQL обусловлен несколькими факторами.

## ACID

PostgreSQL полностью поддерживает ACID-транзакции.

Это гарантирует:

- атомарность операций;
- согласованность данных;
- изоляцию транзакций;
- сохранность информации после сбоев.

---

## Отличная интеграция с Django

Django изначально ориентирован на PostgreSQL.

Используются возможности:

- JSONField;
- полнотекстовый поиск;
- индексы;
- агрегатные функции;
- транзакции.

---

## Производительность

PostgreSQL эффективно работает с:

- большим количеством пользователей;
- сложными JOIN;
- фильтрацией;
- сортировкой;
- агрегацией.

---

## Масштабируемость

В дальнейшем возможно использование:

- репликации;
- партиционирования;
- read replicas;
- логической репликации.

# Архитектура базы данных

База данных состоит из нескольких логических областей.

```
Users
│
├── Accounts
├── Profiles
├── Skills
├── Offers
├── Reviews
└── Notifications
```

Каждая область соответствует отдельному Django-приложению.

# Типы связей

В проекте используются три основных типа связей.

---

## One-to-One

```
User

↓

Profile
```

Каждый пользователь имеет один профиль.

Каждый профиль принадлежит одному пользователю.

---

## One-to-Many

```
User

↓

Offers
```

Один пользователь может создать множество предложений.

---

## Many-to-Many

```
Users

↓

Skills

↓

Users
```

Один пользователь может владеть множеством навыков.

Один навык может принадлежать множеству пользователей.

Для хранения используется промежуточная таблица.

---

# Первичные ключи

Во всех моделях используется автоматически создаваемый первичный ключ.

```
id BIGINT PRIMARY KEY
```

Преимущества:

- высокая производительность;
- совместимость с ORM;
- простота работы с внешними ключами.

---

# Внешние ключи

Связи реализуются через ForeignKey.

Пример:

```
Offer

sender_id

receiver_id
```

Использование внешних ключей обеспечивает:

- целостность данных;
- невозможность появления "висячих" записей;
- автоматическое построение JOIN.

---

# Ограничения целостности

Используются следующие ограничения.

## NOT NULL

Обязательные поля не допускают значение NULL.

---

## UNIQUE

Используется для:

- username;
- email;
- уникальных комбинаций полей.

---

## CHECK

Планируется использование для:

- рейтингов;
- статусов;
- уровней навыков.

Например:

```
rating >= 1

rating <= 5
```

---

## Foreign Key Constraints

Предотвращают удаление записей, нарушающих целостность базы данных.

---

# Индексы

Для повышения производительности используются индексы.

---

## Автоматические

PostgreSQL автоматически индексирует:

- Primary Key;
- Unique.

---

## Пользовательские

Планируется индексирование:

```
username

email

skill

status

created_at
```

---

## Составные индексы

Например:

```
(skill, level)

(status, created_at)
```

---

## Частичные индексы

Используются для часто запрашиваемых данных.

Например:

```
WHERE status='ACTIVE'
```

---

# Нормализация данных

База данных проектируется согласно требованиям третьей нормальной формы (3NF).

Основные принципы:

- отсутствие дублирования данных;
- отсутствие вычисляемых полей;
- минимизация избыточности;
- использование связей вместо копирования информации.

---

# Работа через ORM

Все операции выполняются через Django ORM.

Преимущества:

- безопасность;
- защита от SQL Injection;
- переносимость;
- читаемый код.

Использование Raw SQL допускается только при наличии серьёзных проблем производительности.

---

# Оптимизация запросов

Для повышения производительности используются:

- select_related()
- prefetch_related()
- annotate()
- aggregate()
- exists()
- only()
- defer()

При необходимости производится анализ через

```
EXPLAIN ANALYZE
```

---

# Транзакции

Все критически важные операции выполняются внутри транзакций.

Например:

```
Создание предложения

↓

Изменение статуса

↓

Создание уведомления
```

Если одна из операций завершится ошибкой, все изменения будут отменены.

Для этого используется

```
transaction.atomic()
```

---

# Миграции

Изменение структуры базы данных выполняется исключительно через Django Migrations.

Используются команды:

```
makemigrations

migrate
```

Ручное изменение структуры базы данных запрещено.

# Правила проектирования моделей

При разработке моделей используются следующие соглашения.

- понятные имена таблиц;
- понятные имена полей;
- использование ForeignKey вместо хранения идентификаторов;
- обязательные поля `created_at`;
- обязательные поля `updated_at`;
- использование `verbose_name`;
- использование `related_name`;
- минимизация nullable-полей.

# Заключение

База данных SkillBridge проектируется с использованием современных подходов к построению реляционных систем хранения данных. Основное внимание уделяется целостности информации, производительности запросов и возможности дальнейшего масштабирования.

Использование PostgreSQL совместно с Django ORM позволяет получить надёжную и расширяемую основу для реализации всей бизнес-логики проекта.
